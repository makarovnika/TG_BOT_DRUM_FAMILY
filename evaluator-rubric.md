# Evaluator Rubric

Используй эту рубрику **после реализации** фичи и **до её финальной приёмки**
(переход в статус `passing` в `feature_list.json`). Заполняется ровно одна
таблица на одну фичу. Сохраняй заполненный экземпляр в `evidence` фичи —
например, как комментарий к коммиту или отдельный файл `evidence/<feature_id>.md`.

| Категория | Вопрос | Оценка (0–2) | Заметки |
| --- | --- | --- | --- |
| Correctness | Соответствует ли реализованное поведение `user_visible_behavior` фичи? |  |  |
| Verification | Все ли пункты `verification` реально были запущены, есть ли доказательства? |  |  |
| Scope discipline | Сессия осталась в рамках выбранной фичи, без посторонних правок? |  |  |
| Reliability | Результат переживает перезапуск/повторный прогон без ручного ремонта? |  |  |
| Maintainability | Код и документация достаточно понятны для следующей сессии? |  |  |
| Handoff readiness | Сможет ли свежая сессия продолжить работу, опираясь только на артефакты репо? |  |  |

**Шкала:**
- `0` — не выполнено
- `1` — выполнено частично / есть оговорки
- `2` — выполнено полностью

**Порог приёмки:** все категории ≥ 1, суммарно ≥ 9 из 12.
При оценке `0` хотя бы в одной категории — статус остаётся `in_progress` или
переводится в `blocked`.

## Verdict

- [ ] Accept — фича уходит в `passing`, evidence обновлено
- [ ] Revise — нужны точечные правки, фича остаётся `in_progress`
- [ ] Block — фичу нельзя продвинуть без внешнего решения (см. ниже)

## Required Follow-Up

- Missing evidence: что именно надо добавить в `evidence`
- Required fixes: список конкретных правок до следующего ревью
- Next review trigger: какое событие запускает повторное ревью (например,
  «после получения ответа от поддержки YClients по SMS-подтверждению»)
